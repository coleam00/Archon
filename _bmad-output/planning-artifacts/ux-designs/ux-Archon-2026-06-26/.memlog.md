topic: Route Loop Routing UX
updated: 2026-06-26T20:39

- (decision) Use fast path from the finalized Route Loop Routing PRD and addendum. Source scope is production Web workflow builder plus workflow run detail observability, with secondary builder surfaces treated as block or full round trip.
- (decision) Inherit the existing Archon Web visual system: dark-only shadcn and Tailwind tokens, React Flow canvas, compact node cards, fixed graph dimensions, type badges, left color stripe, metadata pills, and dense utility panels.
- (decision) Represent route_loop as a distinct controller node, not as the existing AI loop. The node must show one target input and three labeled source ports: positive, negative, exhausted.
- (decision) The Route Loop inspector exposes condition, max_iterations, from, and route target fields. Input edge editing remains the preferred way to set from and depends_on, and the inspector mirrors the synchronized value rather than allowing drift.
- (decision) Builder save and run remain blocked when any required route target is missing or when the input edge cannot synchronize route_loop.from and depends_on. Validation copy must name the route outcome and node ID.
- (decision) Run detail graph distinguishes not_activated, pending, skipped, running, completed, failed, and cancelled where applicable. Not activated means unselected in route-aware state and must not be shown as skipped.
- (decision) Run detail exposes route decisions through node_routed event rows and selected route edge styling. Each route decision shows outcome, target, condition, condition_result, negative_count, max_iterations, attempt, and execution sequence when available.
- (decision) Main run summary remains latest-attempt-only. Attempt history and older route decisions live in event detail and logs, with compact attempt chips on nodes where the latest attempt is not the first.
- (decision) Route Loop controllers are not directly retryable. Retry affordance copy for nodes inside a Route Loop says selected route path instead of descendants when route-aware retry semantics apply.
- (decision) Route Loop visual language uses a route-specific teal-green token for the controller stripe and port accents. Outcome states use existing semantic tokens: success for positive, warning for negative, error for exhausted, and muted border for not_activated.
- (decision) Accessibility floor for Route Loop includes keyboard-operable node creation, edge creation, inspector edits, validation navigation, route-event navigation, visible focus rings, and non-color labels for every route outcome and node state.
- (assumption) No Figma, screenshots, or brand deck were provided. The UX handoff therefore anchors on the finalized PRD, addendum, current Web UI conventions, and one static key-screen mock.
