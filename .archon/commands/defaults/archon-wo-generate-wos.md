# Generate Work Orders

**Original idea**: $ARGUMENTS

**Foundation answers**: $foundation-gate.output

**Approved approach**: $approach-gate.output

**Approved decomposition**: $decomposition-gate.output

---

## Your Role

You are generating production-quality BKM Work Orders from the approved plan.
Each WO must be self-contained and complete enough for autonomous implementation
without any additional context.

---

## Phase 1: SETUP

Create the output directory:

```bash
mkdir -p $ARTIFACTS_DIR/work-orders
```

---

## Phase 2: VERIFY CODEBASE STATE

Before generating WOs, do a final codebase check to ensure all file paths are accurate:

1. **Use CGC/Neo4j** (if available) to verify dependency relationships between files
   that will appear in "Files to Read First" sections — ensure the graph reflects
   current state, not stale assumptions from earlier phases
2. **Verify every file you plan to reference actually exists**:
   ```bash
   ls -la {path/to/file}
   ```
3. **Read the relevant files** — confirm line numbers, function names, type names
4. **Check git log** — `git log --oneline -5` for any recent changes

---

## Phase 3: GENERATE EACH WORK ORDER

For each WO in the approved decomposition, generate a file at:
`$ARTIFACTS_DIR/work-orders/WO-{NNN}-{kebab-slug}.wo.md`

Use this BKM template exactly:

```
# WO-{NNN}: {Title}

**Priority**: P{0|1|2|3}
**Effort**: {XS|S|M|L|XL}
**Domain**: {packages/workflows | packages/core | packages/adapters | packages/server | packages/web | packages/cli | packages/git | packages/isolation | packages/providers | packages/paths}
**Conductor Track**: {Schema & Types | Business Logic | API Routes | UI Components | CLI Commands | Tests | Config & Defaults}

---

{2-4 sentences: why this WO exists, what problem it solves, how it fits in the
overall plan from the approved decomposition. Reference prior WO outputs if dependent.}
