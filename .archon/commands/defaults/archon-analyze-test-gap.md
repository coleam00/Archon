# Analyze Test Gap

A bug escaped the test suite. Your job is to figure out why, and
write the spec for fixing the test layer so this class of defect is
caught next time. You do not write tests yourself; the
`archon-repair-tests-from-review` agent does that next.

## Inputs

- `$ARTIFACTS_DIR/contract.md` — canonical interface contract for
  the affected files (just generated by `archon-contract-author`).
- `$ARTIFACTS_DIR/groomed-bug.json` — the verdict from the groom
  phase, including reproduction steps, expected vs actual,
  affected files, nearby tests, and the extrapolated ACs.
- `$ARTIFACTS_DIR/task-context.md` — original ticket text.
- The existing test files in `nearby_tests[]` from the groomed
  bug. Read them.
- The production source in `affected_files[]` from the groomed
  bug. Read it.

## Mission

Classify the test-coverage gap and write
`$ARTIFACTS_DIR/test-gap-analysis.json` listing exactly what the
test-repair agent must do.

## Phase 1: CLASSIFY — Why did this slip through?

For each AC in the groomed bug, classify the gap into one of:

- `no_test` — no existing test covers this AC's behavior. The
  affected production path is exercised by no test at all.
- `weak_test` — a test covers the path but the assertions don't
  catch the defect. Example: test asserts the function returned
  *something*, but doesn't assert *what*.
- `wrong_assertion` — a test makes a positive claim that's
  false on the bug, but the test is currently passing because
  of a mock that hides the failure (mocked dependency returning
  always-success, fixture that masks the input shape, etc.).
- `untested_path` — a test exists for the happy path but the
  bug is on an edge case the test doesn't exercise (null input,
  empty array, concurrent call, race, etc.).
- `test_was_correct` — the test layer would have caught this
  but a different test was disabled/skipped/removed. Names the
  removed test.

Each classification has a different repair strategy. Mark each AC
with its classification.

## Phase 2: SPECIFY — What the repair agent must do

For each `acceptance_criteria` entry, write a `repair_spec`:

- `target_file` — which test file gets edited (or `new` for a
  net-new test file)
- `target_test_name` — the name of the new or edited test
- `gap_classification` — from Phase 1
- `existing_test_to_modify` — file:line of the existing test, if
  this is a `weak_test` or `wrong_assertion` repair
- `required_assertions[]` — exact behavior the test must verify.
  Each assertion is a single observable outcome. Examples:
    - `expect(result).toEqual({ status: "draft", body: "" })`
    - `expect(saveDraftMutation).not.toHaveBeenCalled()`
    - `expect(screen.getByText(/draft is empty/i)).toBeVisible()`
- `selector_or_signature_source` — point at the contract entry
  (file/export/signature) that determines what selector or
  function shape the test should use. Tests must align with the
  contract; if the contract is silent, flag it as
  `contract_inadequate` and the workflow will regenerate.
- `red_state_expectation` — predict what error the test will
  produce against current production code. The next phase
  asserts this prediction holds. Examples:
    - `assertion error: expected toast "Draft is empty" but no toast rendered`
    - `unhandled exception: TypeError: Cannot read property 'body' of undefined`
    - `assertion error: saveDraftMutation was called once, expected 0 times`

A repair spec missing `red_state_expectation` is invalid — the
workflow's next phase needs that prediction to verify the test
actually catches the bug.

## Phase 3: REVIEW — Cross-check against the contract

For every `repair_spec`:
- Does the test target a file the contract names? If not, halt
  with `contract_inadequate: true` so the contract regenerates.
- Does the assertion shape align with the contract's
  `signatures:` and `queries_used:`? If the contract says the
  function returns `Promise<DraftState>` and the test asserts a
  synchronous return, the contract or the test is wrong — flag
  it.
- Are selectors derived from the contract's named visible text /
  aria-labels? Tests that invent selectors will not survive a
  reasonable production implementation.

## Output Schema

Write `$ARTIFACTS_DIR/test-gap-analysis.json` as valid JSON with
this shape:

```json
{
  "summary": "one paragraph: why did the bug escape, what's the repair shape",
  "contract_inadequate": false,
  "contract_gaps": [],
  "repair_specs": [
    {
      "acceptance_criterion": "AC1: ...",
      "gap_classification": "no_test | weak_test | wrong_assertion | untested_path | test_was_correct",
      "target_file": "tests/wor-NN/...",
      "target_test_name": "specific test name",
      "existing_test_to_modify": "tests/...:42 | null",
      "required_assertions": ["assertion 1", "assertion 2"],
      "selector_or_signature_source": "contract section / file / export",
      "red_state_expectation": "exact failure message the test will produce against current production code"
    }
  ]
}
```

## Success Criteria

- **EVERY_AC_COVERED**: Every `acceptance_criteria` from
  `groomed-bug.json` has at least one `repair_spec` entry.
- **RED_STATE_PREDICTED**: Every `repair_spec` has a non-empty
  `red_state_expectation`. This is required to verify the
  tests catch the bug.
- **CONTRACT_ALIGNED**: Every assertion shape and selector is
  traceable to the contract. If not, set
  `contract_inadequate: true`.
- **NO_EDITS**: No files were modified by this analysis.
