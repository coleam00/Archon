## Codebase Patterns
- Web route-loop rendering should use local structural guards while `packages/web/src/lib/api.generated.d.ts` lags the server schema; avoid importing `@archon/workflows` into `@archon/web`.
- Route-loop Web validation mirrors the engine contract in two layers: `validation/structural.ts` owns safe node id and reserved key checks, while `validation/graph.ts` owns one-input, required-route, known-target, and `from` mismatch checks.
- Web route decisions must flow through both live SSE state (`DagNodeEvent.routeDecision`) and REST event projection (`buildWorkflowDagNodeStates`) so `node_routed` survives a one-cycle refresh.
- React Flow route-loop edges use the outcome as `sourceHandle` and `label`; keep dependency edges unchanged and add duplicate-id guards when multiple outcomes target the same node.
- `workflowEventRowSchema` now validates `node_routed` data with `routeLoopDecisionEventDataSchema`; server wire schemas that narrow this refined schema must use Zod `safeExtend()`.
- `serializeSafeCondition()` in `packages/workflows/src/condition-evaluator.ts` is the canonical route-loop condition redaction path; `route-loop-state.ts` delegates to it before persisting route decisions.
- Adding a new `WorkflowEmitterEvent` variant requires exhaustive handling in the Web bridge, Slack bridge, and persisted dashboard row mapping so live refresh paths stay type-safe.
- Route-loop debug surfaces should preserve raw historical events while projecting latest node state separately; API tests should assert both `events` history and `nodeStates` latest projection.
- Route-loop execution now has a minimal route-activation path for the US2 TDD flow: negative routes can re-enter the selected target, positive routes activate only the success target, and exhausted routes remain completed control-flow outcomes.
- Route-loop runtime transitions are centralized in `packages/workflows/src/route-loop-state.ts`; use `applyRouteLoopTransition()` for negative counters, positive resets, route-loop attempts, execution sequence, route activation metadata, route-loop output, and safe condition redaction.
- Durable route-decision persistence now has a core DB transaction helper in `packages/core/src/db/workflows.ts`; future store-adapter work should wire route decisions through `persistRouteDecisionTransition()` instead of generic `updateWorkflowRun()` plus best-effort event writes.
- Console builder round-trip and validation baselines pass before route-loop UI changes, so future builder failures are likely from route-loop serialization or validation work.
- Route-loop workflow fixtures in `packages/workflows/src/test-utils.ts` intentionally return raw workflow-like objects until `dagNodeSchema` supports `route_loop`; do not parse them as `WorkflowDefinition` before the schema phase.
- Route-loop runtime metadata validation lives in `packages/workflows/src/schemas/workflow-run.ts` and validates only `loopCounters`, `nodeAttempts`, `executionSeq`, and `routeActivations` while preserving unrelated metadata keys.
- Route-loop authoring schemas now live in `packages/workflows/src/schemas/route-loop.ts`; loader validation uses the shared safe node-id grammar, while runtime execution still fails fast until executor routing tasks land.
- Use `extractConditionReferences()` from `packages/workflows/src/condition-evaluator.ts` when loader or validation code needs to inspect condition node refs; do not duplicate the condition grammar.
- Route-loop loader validation treats route edges as runtime edges layered on top of normal dependencies: only negative routes may re-enter `route_loop.from`, positive and exhausted routes must be exit paths, and retry paths must be self-contained unless negative directly targets `from`, which is warn-only.
- Console builder wire compatibility belongs in `packages/web/src/experiments/console/builder/types/wire.ts`; do not import server packages into Web when generated API types temporarily lag schema changes.
